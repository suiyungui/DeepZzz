## 运行链路

### 视频链路

```text
USB 摄像头
  -> devices.camera.CameraPipeline
  -> ffmpeg 单进程采集
     -> 高清 HLS 预览：runtime/hls/stream.m3u8
     -> 低清分析帧：MJPEG pipe
  -> pipelines.video_pipeline.VideoPipeline
  -> ai.vision_ai.VisionWorker
  -> ai.pose.YoloPoseEngine
  -> network.edge_http
     -> /hls/*
     -> /api/detections
     -> /api/snapshot
     -> /api/status
```

### 音频/哭声链路

```text
USB 麦克风
  -> devices.mic.ContinuousWaveformRecorder
  -> arecord 持续采集 PCM
  -> pipelines.audio_pipeline.AudioPipeline
  -> 滑动窗口读取最近 yamnet_seconds 秒
  -> ai.audio_ai.YamnetCryDetector
  -> ai.audio_ai.CryStateTracker
  -> network.edge_http
     -> /api/status
     -> /api/audio/yamnet
```

### 昼夜/IR-CUT 链路

```text
day-night-sync.service
  -> scripts/day_night.py sync-ircut
  -> devices.day_night_gpio
  -> GPIO35 读取昼夜状态
  -> devices.ircut
  -> scripts/ir_cut_control.py
  -> GPIO33/GPIO46 控制 IR-CUT
```

网页昼夜状态：

```text
services.state
  -> devices.day_night_sensor.day_night_status()
  -> 优先读取 runtime/state/day_night
  -> fallback 到 runtime/logs/sync_day_night_ircut.log
  -> fallback 到 runtime/state/current_mode
```

## `devices/`

硬件采集和硬件控制边界。原则：只负责拿数据或控制 GPIO，不写业务判断。

| 文件 | 作用 |
| --- | --- |
| `devices/__init__.py` | Python package 标记。 |
| `devices/camera.py` | 摄像头管线。启动 ffmpeg，一个进程同时生成 HLS 和低清分析帧。 |
| `devices/mic.py` | 麦克风录音。包含持续 PCM 采集、一次性录音、音频电平和 wav 写入。 |
| `devices/day_night_gpio.py` | GPIO35 昼夜输入读取，以及联动 IR-CUT 的 watch/sync 逻辑。 |
| `devices/day_night_sensor.py` | 给主网页读取昼夜状态。优先读同步服务写入的状态文件，失败后读同步日志和 IR-CUT 状态文件。 |
| `devices/ircut.py` | IR-CUT H 桥 GPIO 控制。负责 GPIO33/GPIO46 脉冲输出。 |

## `ai/`

AI 推理边界。原则：只关心模型和算力，不关心数据来源和输出方式。

| 文件 | 作用 |
| --- | --- |
| `ai/__init__.py` | Python package 标记。 |
| `ai/runtime.py` | CPU/NPU provider 边界。负责 `SpaceMITExecutionProvider` 注册、provider 顺序和 CPU fallback。 |
| `ai/pose.py` | YOLOv8-pose ONNX 推理、预处理、后处理和骨架绘制。通过 `ai.runtime` 创建 ONNX Runtime session。 |
| `ai/vision_ai.py` | 视觉 worker。只保留最新待处理帧，避免 backlog，统计实际分析 fps。 |
| `ai/audio_ai.py` | YAMNet 推理、连续 PCM 环形缓冲、wav 读写、哭声标签评分和状态平滑。 |

## `pipelines/`

业务流水线层。负责把硬件输入、AI 推理和服务状态串起来。

| 文件 | 作用 |
| --- | --- |
| `pipelines/__init__.py` | Python package 标记。 |
| `pipelines/video_pipeline.py` | 视频流水线。启动摄像头、HLS、低清分析帧和视觉 worker。 |
| `pipelines/audio_pipeline.py` | 音频流水线。启动持续录音、滑动窗口 YAMNet 检测和哭声状态平滑。 |

## `services/`

长期运行状态层。负责协调各条 pipeline，并生成统一状态。

| 文件 | 作用 |
| --- | --- |
| `services/__init__.py` | Python package 标记。 |
| `services/state.py` | 核心运行状态 `EdgeState`。协调视频、音频、资源、昼夜状态，并生成 `/api/status` 数据。 |
| `services/resources.py` | 资源监控。Linux 上读取 CPU、内存、温度、`main.py`/ffmpeg 占用；Windows 本地测试时返回 unavailable。 |

网页的 NPU 占用当前是估算值：`最近一次 NPU 推理耗时 * 实际分析 FPS / 1000`。板端当前没有稳定暴露真实 NPU 硬件利用率的 sysfs/debugfs 节点，因此不显示伪造的硬件计数器。

## NPU 移植状态

NPU 相关逻辑统一放在：

```text
ai/runtime.py
```

当前板端已验证：

| 模型 | Provider | dummy 推理耗时 |
| --- | --- | --- |
| `/home/z/.brdk_models/pose/yolov8n-pose-320.onnx` | `SpaceMITExecutionProvider` + CPU fallback | 约 936 ms |
| `/home/z/.brdk_models/pose/yolov8n-pose-320.onnx` | `CPUExecutionProvider` | 约 1043 ms |
| `/home/z/.brdk_models/pose/yolov8n-pose.q.onnx` | `SpaceMITExecutionProvider` + CPU fallback | 约 170 ms |
| `/home/z/.brdk_models/pose/yolov8n-pose.q.onnx` | `CPUExecutionProvider` | 约 1889 ms |

当前默认配置使用：

```text
POSE_MODEL=/home/z/.brdk_models/pose/yolov8n-pose-320.onnx
POSE_PROVIDER=CPUExecutionProvider
POSE_PROVIDER_FALLBACK=0
```

后续建议：

1. 用真实摄像头画面验证关键点输出是否和 CPU FP32 一致。
2. 连续运行 10-30 分钟观察 NPU 稳定性、CPU 占用、温度和服务重启情况。
3. 如果姿态关键点稳定，再把 NPU 配置固定为生产默认；如果不稳定，保留 fallback 并回退 CPU FP32 模型。
